Multi-Agent RAG over RAMDocs

Architecture driven by measurement, not taste.
Evaluated on a curated 12-question RAMDocs subset that spans every conflict category (pure correct · misinfo · noise · mixed). The same harness scales to the full set — see §10.
Research report on agentic RAG architectures · May 2026 · by Andrey Muruntaev · LinkedIn · Source on GitHub

1 · The task

RAMDocs

A RAG benchmark where retrieved documents deliberately conflict.

  • correct — gold-aligned
  • misinfo — synthetic fabrications
  • noise — off-topic distractors

Multi-answer

Most questions admit 2–3 valid gold answers due to entity ambiguity (homonyms).

  • e.g. Without You I'm Nothing — Placebo / Sandra Bernhard
  • e.g. Doug Harvey — Hockey / Baseball

Constraints

  • Retriever + Analyzer + Mediator required
  • Citations & per-step traces
Goal of this report. Not a single working solution — a justified architecture choice: what alternatives were considered, why this one, how it was measured, and what was improved between versions.

2 · Conflict-sensitive RAG — four families of approaches

FamilyRepresentative workWhy notPick
Training-based Self-RAG (Asai 2024), RAFT (Zhang 2024) requires fine-tuning → out of budget ✗
Single-agent corrective CRAG (Yan 2024) one prompt averages over the entire pool ✗
Self-verification Chain-of-Verification (Dhuliawala 2024) no retrieval — does not resolve source conflicts ✗
Multi-agent MADAM-RAG (Wang 2024), Multi-Agent Debate (Du 2023), Astute RAG (Google 2024) — ✓ chosen
Why MADAM-RAG specifically. Same paper as RAMDocs → direct benchmark anchor · per-document analysis is the structural defence against majority-misinfo · no fine-tuning required, fits the brief.

My modification. 3-round debate loop → one round + deterministic vote + LLM-mediator on disagreement. Hence v1 is called MADAM-lite.

3 · Versioned project structure

Three independent layers

Measurement stand — write once, never edit

11 deterministic metrics · run-registry · immutable artefacts dump. No LLM-judge: RAMDocs ships per-doc ground truth.

Versioned pipelines — interchangeable implementations

One folder per version: pipelines/v1_0_madam_lite/, v2_0_entity_first/, v3_3_analyzer_tuned/, v4_1_promptfix/.

Immutable run artefacts — frozen for the record

runs/<version>/<ts>/ → config.yaml · trace.jsonl · metrics.json. Never edited after freeze. Bug found → new minor version.

Freeze rule. Each frozen version runs twice consecutively. If the two runs diverge by more than ±1 answer out of 12 — the version is not frozen; the prompt is brittle or the code is non-deterministic, dig in.

Side branch v2.1 — calibrated confidence (rejected)

What was tried. Before freezing v2.0, an attempted minor iteration with calibrated confidence in the analyzer + grouping threshold 0.50 (instead of 0.40) — aimed at restoring misinfo-rejection.

What happened. Misinfo +0.04 and Citation +0.04, but at the cost of EM −0.08 and Recall −0.04. Net trade-off not worth it — branch rejected, kept in repo as a frozen regression with the note: «prompt-only misinfo fix is hitting a ceiling — needs an architectural fix». That is the explicit justification for jumping into v3 (Skeptic).

4 · v1.0 · MADAM-lite baseline

Porting the legacy logic into the new harness without architectural changes should reproduce its measured behavior: EM-substring ≈ 0.83 and misinfo-rejection ≥ 0.90. The goal is to lock down a reference point against which every future version will be measured.

✅ Strengths

  • Misinfo-rejection 0.917 overall; in mixed_conflict it sits at a perfect 1.00.
  • Precision-answers 0.833 — virtually no spurious or invented answers.
  • Correct citation 0.917 — the cited documents are almost always the correct-typed ones, not misinfo or noise.
  • Reproducibility: 11 / 12 questions produce identical answers across both freeze runs.
  • Cost: $0.05 for a 12-question run — negligible relative to the insight gained.
  • Transparency: the deterministic vote handles the easy cases; the LLM mediator is invoked only when it's actually needed.

⚠ Weaknesses

  • Recall-all-gold 0.59 — the structural cap of single-answer architecture. On questions with 2–3 valid gold answers we surface at most one.
  • pure_correct is the worst category by F1 (0.33). Paradoxically the "clean" subset is full of homonyms (e.g. Without You I'm Nothing — Placebo / Sandra Bernhard), and v1.0 picks just one — half the gold is lost.
  • Noise rejection 0.50 — half the noise docs remain silent rather than being explicitly rejected. Not catastrophic (faithfulness still holds), but worth restoring in v2+.
  • Abstention 0.00 — there's no "I don't know" pathway. On a degenerate retrieval pool the model will still answer.
  • Coverage 0.42 — fewer than half of the useful documents are used. The mediator picks the strongest evidence rather than aggregating it.
  • EM-any-gold is a weak metric here — it penalises the correct multi-answer format "X | Y". Do not use it as an anchor.
Bottom line: v1.0 handles misinfo and noise well, but ambiguity is structurally beyond reach. Nine out of twelve RAMDocs questions admit ≥ 2 valid gold answers; a single-answer architecture systematically discards half of that. This is the motivation for v2.0 (entity-first decomposition).

5 · v2.0 · Entity-First Decomposition

The Recall@gold = 0.59 cap of single-answer v1.0 will break once the pipeline returns multiple answers grouped by entity. Target: Recall ≥ 0.75, F1 ≥ 0.75, EM lifted by a bare-answer format, and misinfo-rejection holding flat.

✅ Strengths

  • EM-any-gold 0.833 — the bare-answer rule plus strict disambiguation lift this from 0.33. EM = 1.00 across every question in has_misinfo, has_noise and mixed_conflict.
  • Recall@gold +0.077 — the single-answer structural cap is broken. Ambiguous questions like Without You I'm Nothing now return both valid entity variants.
  • Noise rejection +0.264 — entity grouping implicitly closes the second hole. A document without an entity never enters any group and lands in rejected by construction.
  • Coverage +0.131 — more correct documents are exposed as supporting (each variant ships all docs of its entity group).
  • Cost −55% — deterministic grouping plus intra-group voting handles most cases, so the expensive gpt-4o mediator is invoked far less often.
  • Stability: only 2 of 12 questions diverge between freeze runs, and those diverge on capitalization / dedup only.

⚠ Weaknesses

  • Misinfo-rejection −0.125 — a structural side effect: a misinfo document with its own (fabricated) entity forms a standalone group, passes the 0.40 threshold and lands inside its own variant's supporting set. This is not a code or prompt bug — it is an information bound: the analyzer sees a single document and cannot cross-check the entity against the corpus. The fix lives in v3.0 (skeptic).
  • Citation faithfulness −0.122 — under multi-answer, a variant for entity X may cite a document whose eval_metadata.answer is a close-but-different entity.
  • pure_correct F1 = 0.29 — the weak spot is the category itself: questions such as q094 contain misinfo-style texts labeled as correct in RAMDocs (an internal dataset inconsistency). Not addressable from our pipeline.
  • Precision-answers −0.086 — structurally inevitable: emitting more variants opens room for a spurious one.
  • Abstention rate = 0 — there is no "I don't know" pathway even on a degenerate retrieval pool. Targeted by v5.0.
Bottom line: entity-first does break the structural cap of single-answer architecture — EM jumps by +0.50, Recall by +0.077, Noise-rejection by +0.264, and cost drops by 55%. The price is a 0.125 drop in misinfo-rejection: a misinfo document with its own fabricated entity now forms a group that looks reliable to the rest of the pipeline. This is an information bound — the analyzer sees one document at a time and cannot verify the entity against the corpus. The fix is the skeptic agent in v3.0, which sees every entity group and validates each one against the rest of the corpus.

6 · v3 · Skeptic verification agent

The misinfo-rejection regression of v2.0 (0.792, down from v1.0's 0.917) is structural: the analyzer sees only one document at a time and cannot cross-check a fabricated entity against the rest of the corpus. Adding an adversarial Skeptic that does see the full pool — and is explicitly taught to be conservative on legitimate homonyms but aggressive on thin-stub misinfo — should recover misinfo-rejection without sacrificing the v2.0 EM and recall gains.

✅ Strengths

  • Misinfo-rejection +0.083 — back to 0.875, the depth-asymmetry rule catches q209-style fabrications (one short stub vs multi-paragraph evidence) that v2.0 let through.
  • Best balanced operating point on the 4-way Pareto: beats v2.0 on misinfo, beats v1.0 on EM (+0.46), beats both internal v3 iterations on recall.
  • No abstention — the pipeline-level fallback guarantees that a Skeptic over-rejection never produces an empty answer.
  • Homonym-safe — explicit prompt rule keeps "Doug Harvey hockey" + "Doug Harvey baseball" as separate valid variants.

⚠ Weaknesses

  • Cost +5× per run vs v2.0 — the Skeptic uses the more expensive gpt-4o model on every question.
  • Stability: 2 / 12 questions diverge between freeze runs (q176 dedup, q131 wavers between "film" and "novel") — the Skeptic still has edge-cases on near-equal evidence.
  • Recall slightly below v2.0 (0.646 vs 0.667) — depth-asymmetry occasionally over-prunes borderline-thin but valid variants.
  • Latency +2 s — one extra LLM round-trip per question.
Bottom line: v3 establishes a balanced Pareto operating point. Compared to v1.0 it is a clear win on every quality axis (EM +0.46, F1 +0.01, Recall +0.06, Citation-faith on par). Compared to v2.0 it trades a small EM / Recall drop for a substantial misinfo-rejection gain (+0.083). The cost is the Skeptic's extra gpt-4o call. v3 is the recommended default when robustness matters; v2.0 stays the better choice when raw answer quality and cost are the priorities.

7 · v4 · Evidence Evaluator — recommended default ⭐

v3 successfully recovered misinfo-rejection to 0.875 but capped EM at 0.792 because the Skeptic uses corpus-level signals only (depth, counter-evidence, homonyms). A structured per-document quality signal — independent of other documents — should let the pipeline (a) sharpen reliability ordering inside each entity group and (b) give the Skeptic an additional axis to discriminate fabricated stubs from legitimate short answers, lifting Misinfo-rejection ≥ 0.90 without losing EM or recall. Combined with a top-anchored analyzer prompt that strictly enforces grounding, the Skeptic should make keep / reject decisions on cleanly grounded entity / text annotations rather than on noisy paraphrase.

✅ What worked

  • Structured per-document quality signal lifts everything when grounding is solid. The Evaluator's trust_score + red_flags pay off only after the Analyzer reliably emits grounded entity / text pairs — once that pre-condition is met, every quality anchor moves in the right direction at once (EM-any, F1, Misinfo, Recall, Citation-faithfulness, Coverage).
  • Probe-driven prompt iteration. Failure modes were reproduced on single (question, document) pairs at ~$0.0005 per probe before any full-pipeline run was attempted. The second-pass patch (q458) was diagnosed and verified on probe before the freeze.
  • GOOD/BAD contrasts beat buried rules. Rules that existed in the initial draft and were ignored started working once they had GOOD/BAD examples and lived at the top of the prompt instead of 160 lines deep.
  • Run-to-run agreement on every anchor. Abstention, Misinfo-rejection, Noise-rejection and Correct-citation are bit-identical across freeze1 and freeze2. The +1-question shift on EM is in a single noise category and is within the stability rule.
  • Shared safety layer is free insurance. 94/94 tests green, no quality regression, adversarial injection structurally neutralised.

⚠ Where it's still soft

  • has_noise F1 = 0.729 is the weakest category — one shaky noise question swings ±1 between runs. Lifting this safely requires care: W_TRUST = 0.35 cuts noise but also helps on misinfo, so naive reduction risks re-introducing misinfo regressions.
  • q306 Oklahoma group still open. The intra-mediator occasionally loses on a pure-table sub-line vs main-line lead. Candidate for v4.x: an explicit is_main_line_value signal from the analyzer, or a lead-sentence boost inside agents.resolve_entity_group.
  • Cost / latency are ~2× v3. ~$0.022/Q and ~24 s/Q. If budget is tight, v3 stays the recommended cheap default.
  • Pure misinfo (q131-style) stays at 0.667 inside has_misinfo — the analyzer is clean but the Skeptic still misses one specific misinformation pattern. Same shape as in v3, not regressed.
Bottom line: v4 · Evidence Evaluator is shipped frozen as the recommended default. Both freeze runs hit every pre-declared anchor (Abstention 0.000, Misinfo-rejection ≥ 0.917, Noise-rejection ≥ 0.972, EM-substring ≥ 0.917, F1-multi ≥ 0.77), and the averaged metrics are a clean Pareto-improvement over v3 (the prior recommended default). Cost / latency roughly double v3 (~$0.022/Q, 24 s/Q), so v3 remains the cheap-default option when budget matters more than the last percentage points of quality.

5 new rules in the analyzer prompt

HARD GROUNDING RULE
If the verbatim span with both entity and text can't be quoted from the document → no_answer. Top-anchored, read first.
CITATION-IS-NOT-DESCRIPTION
On review-style documents a critic's quote ≠ the answer. Explicit GOOD/BAD contrasts.
PURE-TABLE FALLBACK
If a document is a bare table without lead-prose → abstain rather than scrape a number.
PLACE-DISAMBIGUATOR MANDATORY
Cities and towns must include state/country in entity, so three different Longtowns don't collapse into one group.
ROLE-QUESTION DISAMBIGUATION
When the question uses a generic role («Administrator general») and the doc names a specific entity («Fontaine Ministry») — that is NOT an entity mismatch.

Full pipeline
13 LLM calls · ~30 s
⇒ 2 iterations / evening
→
agent_probe
1 LLM call · ~2 s · ~$0.0005
⇒ 20 iterations / evening

Runs one agent (analyzer / evaluator) on one (question, document) pair. Supports --dry-run for prompt audit without spend. Made the v4 analyzer rewrite possible — failure modes were reproduced on probe before any freeze run was attempted.

Main takeaway from v4. Architecture and prompt quality are two independent levers. I could have stopped at v3 and called it done — but being able to change the analyzer prompt under a fixed architecture, and watch exactly which component the improvement comes from, is the whole point of the versioned stand.

Pipeline contract (every version implements this)

class Pipeline(ABC):
    name: str          # registry key, e.g. "v4.1_promptfix"
    version: str       # semver-ish, e.g. "4.1.0"

    def __init__(self, llm: LLMClient, config: dict | None = None) -> None: ...

    @abstractmethod
    def run(self, question: Question) -> RunResult:
        """Execute the full pipeline on one question."""

    def describe(self) -> dict:
        """Snapshot of config — dumped to config.yaml at freeze time."""

What lives in runs/<version>/<ts>/

config.yaml

Snapshot of pipeline.describe() — pipeline graph, prompts, model hashes, thresholds. Tells you what the pipeline was at freeze time.

trace.jsonl

One event per question. final_answer.variants[] with answer · confidence · supporting_doc_ids · entity, rejected_doc_ids, and explanation (Skeptic's keep/reject reasoning, in plain text).

metrics.json

All 11 metrics aggregated and broken down by category (pure_correct / has_misinfo / has_noise / mixed_conflict) + cost + latency.

Immutable after freeze. Bug → new minor version. No LangGraph: plain Python control-flow for legibility and easy mocking (early LangGraph prototype kept in _legacy/ramdocs_rag_v0/).

10 · Three takeaways

Measurement, not taste

Every version = hypothesis + two freeze runs + journal entry. The final default falls out of numbers, not opinion.

Negative results are kept

v2.1 lives in the repo as a frozen regression. Without it v3's design has no justification.

Architecture ≠ prompt

Prompt quality is a separate lever from graph structure. Tooling under it (agent_probe, MockLLM, immutable artefacts) is the main takeaway.

Open questions — candidates for v4.x (not blockers)

q306 Oklahoma — main-line vs sub-line

Intra-mediator occasionally picks the 2010 timeline row over the main-line lead-sentence. Candidate for v4.x: an explicit is_main_line_value signal from the analyzer, or a lead-sentence boost in agents.resolve_entity_group.

has_noise F1 = 0.73 — weakest category

Skeptic doesn't always separate uniform noise from a valid alternative. Lowering W_TRUST is risky — it also helps on misinfo, so naive reduction risks re-introducing misinfo regressions.

12 → 500 questions

Stand is ready to scale: same make bench, one full run per version ≈ $10. Eight versions frozen on the 12-question subset (as permitted by the brief).

11 · Stack & engineering decisions

A short tour through the stack — what was picked, why, what the trade-offs are, and what would need to change before this ships to production with real load.

Runtime & control flow

  • Python 3.13 + plain imperative control flow. No graph framework.
  • Why no LangGraph / LangChain: early prototype was on LangGraph (kept under _legacy/). Dropped it because the graph abstraction made step-by-step debugging harder and tests harder to mock. With 4–5 agents we don't yet have a state machine that needs a graph engine.
  • Re-evaluate when: the pipeline gains conditional branches that depend on intermediate state across more than 2 steps, or we need parallel sub-graphs with rejoin semantics. Then LangGraph (or a custom state machine) starts paying off.

Models

  • OpenAI API. Two tiers — gpt-4o-mini for the parallel high-volume roles (Analyzer × K, Evaluator × K) and gpt-4o for the reasoning-heavy single-shot roles (Mediator, Skeptic). Cost & latency are mostly driven by the cheap tier, quality by the expensive tier.
  • Vendor-agnostic via Protocol. All LLM calls go through a single LLMClient Protocol (complete_json(system, user, schema, schema_name) → parsed result). Swap to Anthropic / Bedrock / a self-hosted vLLM is a one-class change, not a refactor across the codebase.
  • Strict JSON output enforced through Pydantic schemas. The model returns a typed object, not free text. Reduces a whole class of bugs at the parsing layer.
  • Production concern: gpt-4o-mini is unstable on bounded-float ratings — that's exactly what bit v4.0 (Evaluator's trust scores scattered). For a real production deploy I'd put the Evaluator on a fine-tuned smaller model or move it to gpt-4o and re-tune the weight.

Retrieval & embeddings

  • Hybrid BM25 + dense, top-8. BM25 via rank_bm25. Dense via sentence-transformers/all-MiniLM-L6-v2 (~80 MB, bi-encoder, cached in ~/.cache/huggingface). Weights 0.5 / 0.5.
  • Why MiniLM: small, fast on CPU, good enough for the 3–8 doc pools RAMDocs ships. It is not a multilingual or domain-specific model — for a production deploy the swap target is something domain-tuned (BGE, e5-large, or fine-tuned on the company's corpus).
  • No vector DB. RAMDocs questions ship their own candidate pool (3–8 docs each), so we don't index a corpus. For production this is the layer where we plug in pgvector / Qdrant / OpenSearch.

Schemas & contracts

  • Pydantic v2, strict mode. Every cross-boundary object (Question, RAMDoc, Claim, AnswerVariant, FinalAnswer, RunResult, RunMetrics) is a validated model.
  • Effect: contract tests fail loudly on a breaking schema change. Re-running an old pipeline on a new schema (or vice versa) is impossible by construction.
  • Artefact formats: config.yaml (human-readable snapshot of pipeline.describe()), trace.jsonl (one event per question, replayable), metrics.json (aggregates). All append-only and immutable after freeze.

Testing & QA

  • pytest, 94 tests, ~8 seconds, $0 cost. Four layers: unit (pure Python), contract (per-agent I/O with MockLLM), integration (full pipeline with MockLLM), e2e (real OpenAI, opt-in via @pytest.mark.e2e).
  • MockLLM is the key piece — script-driven, deterministic, lets us cover the full graph end-to-end without API spend. It is how «turn around twenty prompt iterations in an evening» is possible at all.
  • Coverage via pytest-cov. Lint & format via ruff.

Observability

  • Langfuse drop-in. If LANGFUSE_PUBLIC_KEY is set in env, OpenAIClient auto-switches to langfuse.openai. Every LLM call lands in the Langfuse UI with prompt / completion / latency / cost. Off by default, free to turn on.
  • Local trace independently written to trace.jsonl regardless of Langfuse — this is the file the per-question diagnostic above is built from. Means the run is fully introspectable from disk, no external service required.

Dev loop

  • Makefile is the entry point: make test, make test-e2e, make bench PIPELINE=v4.1_promptfix, make compare VERSIONS="v3.3 v4.1".
  • agent_probe — custom CLI for isolated single-agent debugging on one (question, doc) pair, ~$0.0005 per call. The single tool that made the v4 analyzer rewrite possible.
  • Pipeline registry — a single side-effect module that imports every version and binds it to a string name. Adding v5 is one file + one register(...) line.

Demo deploy (separate from the core code)

  • Purpose: let a reviewer run the pipeline through a hosted web UI — pick a question, watch the agents work step by step, inspect the full trace — without cloning the repo, installing Python, or paying for an OpenAI key.
  • Lives in demo/, structurally separate from the pipeline. The pipeline is a library; the demo imports it. The demo can be rewritten / replaced / hosted elsewhere without touching a single line of src/ramdocs_rag/.
  • Stack: FastAPI backend + uvicorn behind nginx, running under systemd on a single host. pip install -e . into a venv, no Docker. Adequate for a single-host showcase.
  • UI tracing: the demo uses TracingClient — a subclass of the standard OpenAIClient that captures every LLM call into a context-var buffer so the frontend can render the per-step trace (which agent did what, on which document, with what verdict). One tiny class on top of the existing LLMClient Protocol — zero impact on the pipeline.
  • CI/CD: the demo has its own demo/.github/workflows/deploy.yml (rsync + systemd restart on push). The main pipeline repo intentionally has no CI yet — the demo's workflow does not pull from it directly.
  • Why this matters for production: the pipeline never «cares about a frontend». If we later swap the demo for a real product UI, or expose the pipeline via a different surface (a worker queue, a CLI batch tool, an internal Slack bot), nothing in src/ramdocs_rag/ has to change.

Decisions deliberately NOT taken

  • No LLM-judge. RAMDocs ships per-document ground-truth labels — all 11 metrics are computed deterministically. A judging LLM would add noise, cost, and a second model to maintain. Re-evaluate only if we move to a dataset without per-doc labels.
  • No Docker / no orchestration. Deploy of the demo is pip install -e . into a venv + systemd + uvicorn behind nginx. Fine for a 1-host showcase. For multi-host or autoscaling we'd containerise and move the model client behind a queue.
  • No CI/CD in the main repo. Only the demo has a GitHub Actions deploy workflow. pre-commit + ruff hook is on the to-do list, not blocking.
  • No fine-tuning. Out of budget and contradicts the goal — show that a versioned multi-agent stack can hit good numbers on a hostile corpus with prompts only.
