v4.0 · Evidence Quality scoring (negative result)

FROZEN · 2026-05-16 · v3 plus a per-document Evidence Evaluator — published as an honest regression
TL;DR — this version is a regression. v4.0 adds a per-doc Evaluator agent that emits a structured DocTrust report (trust_score + red_flags). The hypothesis was that a structured per-doc evidence-quality signal would let the Skeptic prune misinfo more sharply. In practice every quality metric ticked down (EM −0.042, F1 −0.025, Misinfo-rejection −0.042) while LLM calls per question jumped +81% and latency +71%. v3.3_analyzer_tuned remains the recommended default. v4.0 is shipped frozen so that the negative result is reproducible and the design lesson is recorded, not forgotten.

Hypothesis

v3 successfully recovered misinfo-rejection to 0.875 but capped EM at 0.792 because the Skeptic uses corpus-level signals only (depth, counter-evidence, homonyms). A structured per-document quality signal — independent of other documents — should let the pipeline (a) sharpen reliability ordering inside each entity group and (b) give the Skeptic an additional axis to discriminate fabricated stubs from legitimate short answers, lifting Misinfo-rejection ≥ 0.90 without losing EM or recall.

Architecture

NEW — added in v4 · everything else is identical to v3

The Evaluator runs alongside the Analyzer on each retrieved document and emits a structured DocTrust report. Its trust_score is consumed by Reliability scoring (W_TRUST = 0.35), and its red_flags are surfaced to the Skeptic inline with each pool document. Crucially, the Evaluator does NOT see other documents and does NOT compare against external facts — it only judges what one document says about itself (internal consistency, encyclopedic quality, specificity, relevance).

Evaluator system prompt (v4 — NEW)

You are the Evidence Evaluator agent in a multi-agent RAG pipeline.

You receive a question and ONE retrieved document. Your job is to score
that document on four LOCAL quality axes and assemble a composite
trust_score plus a closed-enum list of red_flags. You do NOT see other
documents and you do NOT compare against external facts. Judge only what
this document says about itself.

Four axes, each in [0.0, 1.0]:

  1. internal_consistency — does the document contradict itself? Are
     dates, names and figures coherent within this single document?
  2. encyclopedic_quality — does it read like a curated reference entry
     (paragraphs of contextual prose, named subject, dates, figures),
     or like a one-line assertion / category page / formatting cruft?
  3. specificity — does it contain concrete specifics (dates, places,
     people, figures) or only vague generalities?
  4. relevance — is the document topically aligned with the question?

trust_score is the composite quality in [0.0, 1.0]. Approximate it as
~ 0.30·internal_consistency + 0.30·encyclopedic_quality + 0.25·specificity + 0.15·relevance,
then round to two decimals.

red_flags is a list drawn from this CLOSED enum:
  - "short_stub" — the document is one or two sentences with no context
  - "self_contradiction" — internal facts contradict each other
  - "off_topic" — not about the asked subject
  - "no_specifics" — no dates / places / people / figures
  - "category_page" — looks like a Wikipedia category page or a list
  - "formatting_cruft" — broken markup, headers without body, table residue

Output strictly the JSON object with fields:
  doc_id, internal_consistency, encyclopedic_quality, specificity,
  relevance, trust_score, red_flags.

Analyzer system prompt (v4)

Same as v3.3_analyzer_tuned. The analyzer is unchanged.

Skeptic system prompt (v4)

Same four-check skeptic prompt as v3 (entity grounding · citation faithfulness · counter-evidence · depth-asymmetry · homonyms-aware), with one extra input: each document in the pool now arrives with a per-doc trust score and a list of red_flags from the Evaluator. The Skeptic is allowed to consider these signals when arbitrating thin-stub vs rich-context variants, but the four-check decision logic is otherwise identical.

What works / what doesn't

✅ What's interesting (even though it lost)

  • Architecture is clean — the Evaluator is a pure per-document agent with no cross-doc state. DocTrust is a strict Pydantic model with closed-enum red_flags; the new W_TRUST slot replaces the dormant recency / authority placeholders, so the reliability formula now has a fully populated signal stack.
  • Failure mode is mechanistic, not random — we can point at exactly which design choices regressed which metrics (see "Why it failed"), which is the whole reason to publish negative results.
  • Reproducibility — both freeze runs agree on EM, Misinfo-rej and Noise-rej to four decimals. The regression is stable, not stochastic.

⚠ Why it failed

  • gpt-4o-mini is noisy on bounded-float ratings. Trust-scores on legitimate pure_correct documents scatter across 0.00–0.85, adding noise to reliability instead of order.
  • red_flags overlaps with legitimate brevity. On pure_correct questions (q093, q094, q458) the gold documents are short encyclopedic stubs — the Evaluator tags them short_stub / no_specifics, and the Skeptic uses those flags to attack legitimate variants with the same depth-asymmetry rule that worked against misinfo in v3.
  • W_TRUST = 0.35 is too aggressive. It shifts intra-group ranking toward verbose documents — which sometimes coincides with misinfo (RAMDocs fabrications are often longer than correct stubs).
  • Sequential Evaluator calls double the per-doc roundtrip. Latency jumps from 13s to 23s. asyncio.gather would reduce latency but not solve the quality problem.
Bottom line: v4.0 ships frozen as a published negative result. Every quality metric ticked down (EM −0.042, F1 −0.025, Misinfo-rejection −0.042, Citation-faith −0.035) while cost +15% and LLM calls per question +81%. The design lesson is recorded: a per-doc quality agent over a small encyclopedic corpus is not free — it injects LLM rating noise into reliability and gives the Skeptic correlated-with-misinfo flags that also fire on legitimate short documents. v3.3_analyzer_tuned remains the recommended default. A potential v4.1 (drop red_flags, lower W_TRUST to 0.10–0.15, move Evaluator to gpt-4o) is left as a direction, not a commitment.

Unchanged vs v3.3

  • Retriever (BM25 + dense, top_k = 8)
  • Analyzer prompt (the v3.3 multi-candidate disambiguation tune)
  • Intra-Entity Mediator prompt
  • Skeptic prompt body (the four checks); only the input gains trust + red_flags per document
  • Entity grouping logic and the 0.30 weight threshold
  • Mediator / Skeptic models (gpt-4o)
