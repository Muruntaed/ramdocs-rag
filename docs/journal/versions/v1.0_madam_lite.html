Hypothesis

Porting the legacy logic into the new harness without architectural changes should reproduce its measured behavior: EM-substring ≈ 0.83 and misinfo-rejection ≥ 0.90. The goal is to lock down a reference point against which every future version will be measured.

Architecture

A linear pipeline (no LangGraph). Analyzer agents run sequentially — parallelisation is deliberately deferred for the baseline. The mediator only takes the LLM path when the deterministic weighted vote fails to identify a dominant cluster.

Analyzer system prompt

You are the per-document Analyzer in a multi-agent RAG pipeline.

Your job: given a USER QUESTION and ONE DOCUMENT, decide whether the document
provides an answer to the question, and if so extract it.

Output strictly the JSON object matching the `Claim` schema, with these fields:

- `doc_id`        — copy the `doc_id` from the input verbatim.
- `text`          — the candidate answer the document supports. A short noun
                    phrase or sentence (≤ 25 words). Empty string when stance
                    is `no_answer`.
- `stance`        — one of:
                      * `supports`    — the document directly answers the question.
                      * `contradicts` — the document explicitly states the
                                        question's premise is false, or denies
                                        a specific answer it cites.
                      * `no_answer`   — the document is irrelevant, off-topic,
                                        or merely mentions the topic without
                                        addressing the question.
- `confidence`    — float in [0, 1]. How sure are you that `text` is what
                    the document says? Penalise hedged or unclear sources.
- `supporting_quote` — a verbatim span (≤ 200 chars) copied from the document
                       that grounds `text`. Use an empty string when stance is
                       `no_answer`.

Rules:

- Use ONLY the document text. Do not invoke outside knowledge.
- If the document mentions the entity but does not actually answer the
  question, choose `no_answer` (not `supports`).
- If multiple plausible answers appear in the same document, return the one
  the document most clearly endorses; if there is no clear endorsement, choose
  `no_answer`.
- The supporting quote must be a substring of the document, not a paraphrase.

Mediator system prompt

You are the Mediator in a multi-agent RAG pipeline.

Inputs you receive:
- The user QUESTION.
- A list of CLAIMS extracted from individual documents. Each claim carries:
  `doc_id`, `text` (candidate answer), `stance`, `confidence`,
  `supporting_quote`, and a `reliability` score in [0, 1] computed by the
  pipeline from retrieval rank, analyzer confidence, recency and source
  authority.
- A CLUSTERING SUMMARY listing groups of claims that point at the same
  answer, with the summed reliability of each cluster. A weighted-majority
  vote was attempted but was inconclusive (no cluster dominated by the
  required ratio), which is why you were called.

Your job: produce the final answer.

Output strictly the JSON object with these fields:

- `answer` — the answer entity ONLY, as a bare string. No surrounding prose,
  no full sentence, no leading phrase like "The answer is" or "X died on".
  Just the value itself.
    * For a person: "Placebo", not "The artist is Placebo".
    * For a date: "11 March 1707", not "George Horner died on 11 March 1707."
    * For a place: "Glasgow", not "He was born in Glasgow."
  When the question is genuinely ambiguous (e.g., two different entities share
  the same name and both are correct), join both with " | " — e.g. "Placebo |
  Sandra Bernhard", "11 March 1707 | 9 February 1677". Still no prose.
- `confidence` — float in [0, 1]. Lower it when evidence is thin or conflicts
  remain.
- `supporting_doc_ids` — IDs of the documents whose claims you relied on.
- `rejected_doc_ids` — IDs of documents you explicitly judged wrong or
  irrelevant (misinformation, noise, contradicted by stronger evidence).
- `reconciliation_explanation` — 2–4 sentences. Explain which cluster you
  picked, why competing claims were rejected, and how reliability informed
  the call.

Rules:

- Prefer the cluster with the highest summed reliability unless its
  supporting quotes are obviously weaker than a runner-up's. Justify
  deviations in `reconciliation_explanation`.
- Quotes are the strongest evidence. Treat a claim whose `supporting_quote`
  clearly contains the answer phrase as more credible than one without.
- Do not invent answers absent from the claims.
- Never include any `doc_id` in BOTH `supporting_doc_ids` and
  `rejected_doc_ids`.

What works / what doesn't

✅ Strengths

  • Misinfo-rejection 0.917 overall; in mixed_conflict it sits at a perfect 1.00.
  • Precision-answers 0.833 — virtually no spurious or invented answers.
  • Correct citation 0.917 — the cited documents are almost always the correct-typed ones, not misinfo or noise.
  • Reproducibility: 11 / 12 questions produce identical answers across both freeze runs.
  • Cost: $0.05 for a 12-question run — negligible relative to the insight gained.
  • Transparency: the deterministic vote handles the easy cases; the LLM mediator is invoked only when it's actually needed.

⚠ Weaknesses

  • Recall-all-gold 0.59 — the structural cap of single-answer architecture. On questions with 2–3 valid gold answers we surface at most one.
  • pure_correct is the worst category by F1 (0.33). Paradoxically the "clean" subset is full of homonyms (e.g. Without You I'm Nothing — Placebo / Sandra Bernhard), and v1.0 picks just one — half the gold is lost.
  • Noise rejection 0.50 — half the noise docs remain silent rather than being explicitly rejected. Not catastrophic (faithfulness still holds), but worth restoring in v2+.
  • Abstention 0.00 — there's no "I don't know" pathway. On a degenerate retrieval pool the model will still answer.
  • Coverage 0.42 — fewer than half of the useful documents are used. The mediator picks the strongest evidence rather than aggregating it.
  • EM-any-gold is a weak metric here — it penalises the correct multi-answer format "X | Y". Do not use it as an anchor.
Bottom line: v1.0 handles misinfo and noise well, but ambiguity is structurally beyond reach. Nine out of twelve RAMDocs questions admit ≥ 2 valid gold answers; a single-answer architecture systematically discards half of that. This is the motivation for v2.0 (entity-first decomposition).
Run artefacts are immutable: no changes to logic or prompts of v1.0 after freeze. Any change spawns a new minor version (v1.1) with its own freeze run.
